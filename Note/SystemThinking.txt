System
  Is not just a bunch of parts put together but its is a pattern of how those parts affect each other over time
  Change one part and other parts react
  Miss a connection and the pattern breaks
  Let think it's of it like an orchestrator
    System is a music
    AI can play any instrument on demand: violin, guitar, piano...
    Someone has to know how these parts fit together, or when the string should hold back ...
    => that's the conductor
  System thinking comes down with 3 question
    Where does the state live?
      Who own the truth of system?
      If 2 pieces think they own it, you already have a bug
    Where does the feedback live
      What tell you did system work? How do you know?
        Any metric, log, errors
        If nothing tell you this, then system isn't working. It just pretends to work
    What breaks if I delete this?
      Can you trace the blast radius of any component of the system in your head before you touch it?
  4 ways to train your system thinking
    Design before you prompt
      Take a note and draw a box to describe your data flow.
      Mark where state lives, where failure surfaces
      If we can't draw it, we really don't understand and AI will build whatever it want
    Spec as scaffolding
      Write the what and why before AI do the how
      Even with the short spec where you define a problem, a constraints, failure mode ....
    Deletion test
      Run a test where you delete a component and ask yourself how wrong it can be and the blast radius
    Study the generated code
      Don't just accept the code but ask AI to walk you through it
